**"처음부터 끝까지 서비스를 만들어내는 개발자"**

모던 웹 기술 스택을 활용한 풀스택 개발과 클라우드 인프라 구축 경험을 바탕으로,  
안정적이고 확장 가능한 서비스를 구현하는 개발자입니다.

### 1. 설문 조사 플랫폼 (개인 프로젝트)  
> 조사원·응답자·관리자가 한 화면에서 설문을 만들고 이어서 응답할 수 있는 프로토타입

- **기간**: 2025.11 ~ 2025.11  (약 7일)
- **팀 구성**: 1명 (풀스택)  
- **내 역할**: 기획 · 백엔드 · 프런트엔드

#### 사용 기술
**Frontend**: `HTML`, `jQuery`, `Tailwind CSS`  
**Backend**: `NestJS`, `TypeScript`, `Prisma ORM`, `MSSQL`

#### 주요 기능
- 관리자 화면에서 설문 생성·수정·삭제, Excel 업로드/다운로드, 분기 로직 편집, 응답 진행 현황 모니터링
- 응답자 화면에서 설문 목록 조회, 이어하기, 분기 로직 기반 진행률 표시, 문항별 유효성 검증
- NestJS REST API로 설문 구조/응답 저장, CSV 내보내기, Prisma 스키마 기반 MSSQL 연동

#### 내 역할
- 조사원·응답자·관리자 역할을 정의하고 정보 구조와 사용자 흐름 설계
- Prisma 스키마와 마이그레이션 자동화를 통해 타입 안정성 확보 및 DB 스키마 관리

#### 배운 점
- 현장 업무에서 겪은 “이어하기 불가” 문제를 토큰 기반 상태 복원으로 해결하며 UX 중심 사고 강화
- Prisma와 NestJS 조합으로 타입 안정성과 생산성을 동시에 확보하는 백엔드 구조 설계 경험
- 선택지 기반 분기 로직을 표 형태로 정리하고, 경로별 테스트 시나리오를 작성하며 복잡한 흐름을 점검하는 방법 학습

#### 기억에 남는 이슈 해결 or 작업 경험
- **선택지 분기 표 설계**  
  → 복잡한 설문에서 선택지 조합에 따라 문항이 건너뛰거나 반복되는 시나리오 누락이 발생  
  → 설문 흐름을 표로 정리하고 각 선택지를 조건 배열로 정의하여 NestJS 서비스에서 일관된 검증이 가능하도록 정리

#### 회고
> 조사원·응답자·관리자 세 역할을 동시에 만족시키기 위해  
> **정보 구조와 흐름을 처음부터 설계한 경험**이 가장 인상적이었습니다.  
> 단순 CRUD를 넘어 이어하기, 분기 로직, 진행 현황처럼 꼭 필요했던 기능을 직접 만들면서 현장에서 생길 문제를 하나씩 확인하고 고쳤습니다.  
> 응답자의 선택에 따라 설문이 자연스럽게 이어지도록 분기 표를 다듬어  
> 다양한 경로를 포괄하는 안정적인 설문 흐름을 완성할 수 있었습니다.

### 2. Mail Advisor - AI 메일 작성 어시스턴트 (개인 프로젝트)  
> 네이버 메일 작성 시 AI가 실시간으로 전문적인 메일로 재작성해주는 Chrome 확장 프로그램

- **기간**: 2025.10.16 ~ 2025.10.27
- **팀 구성**: 1명 (풀스택)  
- **내 역할**: 풀스택 개발 전담

#### 사용 기술
**Frontend**: `Vanilla JavaScript (ES6+)`, `Chrome Extension Manifest V3`, `Shadow DOM`
**Backend**: `NestJS`, `TypeScript`, `MySQL 8.0`, `TypeORM`, `JWT`, `bcryptjs`  
**AI/LLM**: `OpenAI API (GPT-5-nano)`, `프롬프트 엔지니어링`  
**결제**: `토스페이먼츠 Key-in API`  
**DevOps**: `Docker`, `Docker Compose`

#### 주요 기능
- **AI 기반 메일 재작성**: 프롬프트 엔지니어링을 활용한 컨텍스트 인식 메일 개선
- **실시간 동기화**: iframe 및 Shadow DOM을 활용한 네이버 메일 에디터 실시간 감지
- **8가지 컨텍스트 변수**: 직책, 업무, 톤앤매너, 목표, 수신자, 길이 정책 등 맞춤 설정
- **원클릭 적용**: AI 재작성 결과를 원본 에디터에 즉시 적용 및 되돌리기 기능
- **JWT 인증 시스템**: Access Token(2시간) + Refresh Token(1일) 자동 갱신
- **토큰 기반 과금**: OpenAI 실제 사용량(요청+응답 토큰) 기반 차감 시스템
- **토스페이먼츠 결제**: 토큰 충전을 위한 결제 시스템 통합

#### 내 역할
- 프로젝트 전체 기획 및 풀스택 개발
- OpenAI GPT 모델 활용한 정교한 프롬프트 엔지니어링 설계
- Chrome Extension Manifest V3 기반 확장 프로그램 개발
- Shadow DOM을 활용한 CSS 격리 및 독립적 UI 구성
- JWT 기반 인증 시스템 및 토큰 관리 로직 구현
- 토스페이먼츠 Key-in API 연동 및 결제 시스템 구축
- Docker 컨테이너 기반 백엔드 개발 환경 구축

#### 배운 점
- **프롬프트 엔지니어링**: 단순 API 호출이 아닌 체계적인 instruction 설계를 통해 원하는 출력 형식과 품질 제어
- **Chrome Extension 개발**: Manifest V3 기반 확장 프로그램 개발 및 콘텐츠 스크립트, 백그라운드 워커 활용
- **Shadow DOM 활용**: CSS 격리를 통한 외부 스타일 간섭 방지 및 독립적 UI 구현
- **실시간 DOM 감지**: MutationObserver를 활용한 동적 콘텐츠 변화 추적
- **결제 시스템 통합**: 토스페이먼츠 API 연동 및 결제 프로세스 구현 경험
- **토큰 경제 시스템**: 실제 AI 토큰 사용량 기반 과금 로직 설계 및 구현

#### 기억에 남는 이슈 해결 or 작업 경험
- **네이버 메일 iframe 및 Shadow DOM 접근 문제**  
  → 네이버 메일의 복잡한 iframe 구조와 동적 로딩으로 인해 에디터 요소를 정확히 찾기 어려운 문제 발생  
  → 재귀적 iframe 탐색 로직과 MutationObserver를 조합하여 에디터가 늦게 로드되어도 자동으로 감지하도록 구현  
  → Shadow DOM을 활용하여 네이버 메일의 스타일과 완전히 격리된 독립적인 UI 패널 구축

- **프롬프트 엔지니어링을 통한 출력 품질 제어**  
  → 초기에는 AI가 불필요한 설명, 코드펜스, 제목 제안 등을 함께 출력하여 본문만 추출하기 어려운 문제 발생  
  → 엄격한 출력 제약 규칙과 역할 정의를 instruction에 명시하고, 원문은 input 파라미터로 분리하여 전달  
  → sanitize 함수로 마크다운, 코드펜스, 레이블 등을 제거하는 후처리 로직 추가하여 순수 본문만 반환되도록 개선

- **토큰 사용량 기반 실시간 차감 시스템**  
  → OpenAI API 응답에서 실제 사용된 토큰(input + output)을 추출하여 DB에 즉시 차감  
  → 동시성 제어를 위해 TypeORM의 트랜잭션을 활용하여 토큰 잔액 업데이트의 무결성 보장  
  → 사용자에게 "245 토큰 사용 (남은 토큰: 9755)" 형태로 실시간 피드백 제공

#### 회고
> 이번 프로젝트는 AI 기술을 실제 사용자 경험에 통합하는 과정에서 많은 것을 배울 수 있었습니다.  
> 단순히 OpenAI API를 호출하는 것을 넘어, 프롬프트 엔지니어링을 통해 AI의 출력을 정교하게 제어하고  
> 사용자의 다양한 컨텍스트를 반영한 맞춤형 결과를 생성하는 경험이 가장 인상깊었습니다.  
>  
> Chrome Extension 개발을 통해 브라우저 확장 프로그램의 구조와 제약사항을 이해하게 되었고,  
> Shadow DOM을 활용한 CSS 격리 기법으로 외부 환경과 독립적인 UI를 구축하는 방법을 익혔습니다.  
> 특히 네이버 메일의 복잡한 iframe 구조를 분석하고 동적으로 추적하는 과정에서  
> 실제 서비스 환경에서의 문제 해결 능력을 키울 수 있었습니다.  
>  
> 또한 토스페이먼츠 API를 연동하며 실제 결제 시스템을 구현해본 경험과,  
> 토큰 사용량 기반 과금 로직을 설계하면서 비즈니스 로직과 기술을 연결하는 방법을 배웠습니다.  
>  
> 이 프로젝트를 통해 AI 서비스 개발부터 결제 시스템 통합까지  
> 실제 상용 서비스에 필요한 전체 스택을 경험할 수 있었으며,  
> 특히 프롬프트 엔지니어링과 브라우저 확장 개발이라는 새로운 영역을 개척한 의미 있는 시간이었습니다.

### 3. 스포츠 클래스 예약 시스템 (개인 프로젝트)  
> 스포츠 클래스 신청 및 관리를 위한 풀스택 웹서비스

- **기간**: 2025.06.12 ~ 2025.07.08
- **팀 구성**: 1명 (풀스택)  
- **내 역할**: 풀스택 개발 전담

#### 사용 기술
**Frontend**: `Next.js 15`, `TypeScript`, `Tailwind CSS`, `Axios`  
**Backend**: `NestJS`, `TypeScript`, `MySQL`, `TypeORM`, `Passport.js`, `JWT`, `bcrypt`, `Swagger`  
**DevOps**: `Docker`, `Docker Compose`

#### 주요 기능
- JWT 기반 안전한 회원가입/로그인 시스템
- 스포츠 클래스 탐색 및 카테고리별 검색
- 실시간 클래스 신청 및 상태 확인
- 관리자 대시보드 (신청자 관리, 승인/거절 처리)
- 반응형 디자인 및 모바일 최적화
- 실시간 신청 통계 및 현황 모니터링

#### 내 역할
- 프로젝트 전체 기획 및 풀스택 개발
- 모듈형 아키텍처 설계 (NestJS의 모듈 시스템 활용)
- JWT 인증 시스템 및 보안 구현
- Docker 컨테이너 기반 개발/배포 환경 구축
- 관리자 대시보드 및 사용자 UI/UX 설계

#### 배운 점
- **모던 풀스택 아키텍처**: Next.js 15와 NestJS를 활용한 TypeScript 기반 개발
- **모듈형 설계**: 기능별 독립적인 모듈로 확장성과 유지보수성 확보
- **컨테이너 기반 개발**: Docker 핫리로드와 헬스체크 시스템 구현
- **보안 시스템**: JWT 토큰 인증과 bcrypt 해싱을 통한 안전한 사용자 관리
- **TypeScript 활용**: 런타임 에러 방지 및 타입 안전성 확보



#### 기억에 남는 이슈 해결 or 작업 경험
- **Docker 핫리로드 설정 이슈**  
  → 개발 환경에서 코드 변경 시 자동 재시작이 되지 않는 문제 발생  
  → `CHOKIDAR_USEPOLLING=true`와 `WATCHPACK_POLLING=true` 설정으로 파일 변경 감지 방식을 폴링으로 변경하여 해결

- **다중 관리자 환경에서의 동시성 제어 및 트랜잭션 처리**  
  → 팀원 관리 페이지에서 드래그앤드롭으로 팀원 소개 순서를 변경할 때, 여러 관리자가 동시에 작업하면서 순서 데이터 불일치 문제 발생  
  → TypeORM의 낙관적 락(Optimistic Locking)을 통해 버전 기반 충돌 감지 구현, 동시 수정 감지 시 최신 데이터로 자동 갱신 및 사용자 알림 제공

- **액세스/리프레시 토큰 기반 자동 인증 갱신 시스템**  
  → 액세스 토큰 만료 시마다 사용자가 재로그인해야 하는 불편함과 보안 취약점 발생  
  → 쿠키 기반 리프레시 토큰 저장 및 액세스 토큰 자동 갱신 로직 구현으로 사용자 경험과 보안성을 동시에 확보

>

> 기능별로 완전히 분리된 모듈 구조를 통해 코드의 재사용성과 확장성을 크게 향상시킬 수 있었고,

### 4. LostParty (팀 프로젝트 | 팀 리더)  
> 실시간 게임 파티 정보 제공 웹서비스

- **기간**: 2024.12 ~ 2025.04  
- **팀 구성**: 3명 (백엔드/프론트엔드/OCR)  
- **내 역할**: 백엔드 전담 + 팀 리더 

#### 사용 기술
`Node.js`, `Express.js`, `MySQL`, `Docker`, `AWS EC2`, `Swagger`, `Git`, `Notion`  
프론트엔드 (팀원) : `Next.js` / OCR 모듈 (팀원) : `Python`

#### 주요 기능
- OCR로 게임 화면에서 닉네임 인식 → 블랙리스트 확인
- LostArk API 연동 → 캐릭터 정보 조회
- Swagger를 활용한 API 문서 자동화 및 프론트 협업
- Docker 환경 구성 및 EC2 실서버 배포

#### 내 역할
- 팀 리더 및 백엔드 개발 담당
- 프로젝트 구조 설계 및 전반적인 배포 환경 구성
- 서버 API 구축, AWS EC2 환경 세팅, 도메인 연결, SSL 인증, 로스트아크 API 연동 수행

#### 배운 점
- Docker로 개발환경을 통합하고 실배포까지 경험  
- Swagger를 활용한 API 문서화 및 프론트와의 협업  
- 팀원들의 상황 체크 및 개발 속도 조율로 **리더십 경험 강화**  
- **Git으로 코드 형상 관리 및 기능 분담**, PR 통한 리뷰 및 공유 경험



#### 기억에 남는 이슈 해결 or 작업 경험
- **HTTPS 연결 이슈 (SSL 인증)**  
  → 인증서는 발급되었지만 Nginx 설정 오류로 인해 브라우저에서 “안전하지 않음” 경고가 발생  
  → Certbot을 활용해 SSL 인증서를 재설정하고, `nginx.conf`에서 `ssl_certificate` 경로 및 HTTPS 리다이렉션 설정을 올바르게 구성하여 해결  

- **로컬 개발 환경과 실제 서비스 환경의 차이에서 오는 개념적 어려움**  
  → 그동안 `localhost`에서만 개발하다 보니, “다른 사람은 어떻게 접속하지?”라는 의문과 함께 실제 서비스 배포의 전체 흐름을 이해하는 데 어려움이 있었음  
  → EC2 서버 구축, 포트 개방, 도메인 연결, Nginx 설정, HTTPS 발급 등 웹 서비스의 “운영” 관점에서 필요한 절차를 직접 경험하며 개념을 체득함  

- **Nginx reverse proxy 설정 이슈**  
  → 프론트엔드에서 API 요청이 제대로 전달되지 않거나, 서버 간 충돌이 발생  
  → `/api` 경로 설정 누락, 프록시 포트 중복 문제 등을 반복적으로 점검하며 각 요청 흐름을 정리하고, 명확한 포트 분리와 proxy 설정을 통해 안정화  
#### 회고
> 아이디어에서 시작된 이 프로젝트를 팀 프로젝트로 발전시키면서  
> 다양한 기술과 커뮤니케이션 스킬을 경험할 수 있었습니다.  
> Docker 구성, Swagger API 문서화, 배포 환경 구축,  
> 그리고 팀원과의 협업 경험은 실무와 유사한 소중한 훈련이었습니다.  
>  
> 특히 팀원들이 각자 잘하는 부분(OCR 모델 구성, 프론트 UI 처리 등)을 보며  
> **직접 물어보고, 배우고, 내 것으로 흡수하려고 노력했습니다.**  
> 단순히 맡은 역할만 하는 게 아니라, **서로의 강점에서 배울 수 있는 경험**이 정말 소중했습니다.

> 팀 리더로서 **조율하고 리드하는 능력**을 기를 수 있는 좋은 기회였습니다.

## 기억에 남는 프로젝트
‘로스트파티’는 게임 화면을 실시간으로 인식하여 파티원 정보를 자동 추출하고, 이를 서버로 전달해 정보를 관리·활용하는 시스템입니다. 평소 즐기던 게임의 파티 모집 과정을 자동화하고 싶다는 생각에서 출발해 개인 프로젝트로 시작했습니다.

초기에는 JavaScript 기반 Tesseract.js와 OpenCV.js를 활용하여 클라이언트 OCR 처리를 직접 구현했으나, 한글 및 게임 특수폰트 인식에서 정확도에 한계를 느꼈습니다. 전문적인 OCR 학습과 전처리 기술이 필요하다고 판단해, 관련 기술을 가진 팀원을 직접 모집하고 함께 협업 구조를 설계했습니다.

저는 프로젝트의 기획 및 전체 시스템 구조 설계, 프론트-백 연동 , API 처리 등을 담당했으며, OCR 정확도 향상을 위한 실험과 UI 개선을 주도했습니다. 팀원들과는 역할을 분담하고 주기적인 회의를 통해 목표를 공유하며 개발을 이어갔습니다.

이 프로젝트를 통해 혼자서 끝까지 밀어붙이는 추진력과, 한계를 인정하고 협업을 선택할 줄 아는 유연함, 그리고 협업에서의 커뮤니케이션 감각을 키울 수 있었습니다

## 다른 사람과 구별되는 나만의 강점
저는 문제가 생겼을 때 다양한 관점에서 접근하며, 여러 해결 방안을 스스로 설계하고 그중 최선의 선택지를 찾아 실행하는 능력을 갖추고 있습니다.

프로젝트 ‘로스트파티’를 진행하며 여러 기술적·구조적 문제에 직면했을 때, 단순한 해결보다는 여러 대안을 나열하고 각각의 장단점을 분석한 뒤, 상황에 가장 적합한 방법을 팀원들과 함께 선택해 실행해왔습니다. 예를 들어, 클라이언트 처리 vs 서버 처리, 직접 구현 vs 외부 API 연동 등 여러 갈래의 선택지 속에서 문제해결 방식의 장단점을 균형 있게 고려해 판단했습니다.

이처럼 저는 문제를 복잡하게 만들지 않으면서도 다양하게 바라보고, 해결책을 체계적으로 정리한 후 실행에 옮기는 데 강점이 있습니다. 조직 내에서 ‘기획-판단-실행’을 연결하는 사람으로서 기여할 수 있다고 자신합니다.

## 성장 방향

**"기술과 비즈니스를 연결하는 개발자로 성장"**

현재 풀스택 개발 역량과 클라우드 인프라 경험을 바탕으로, 확장 가능한 시스템 아키텍처를 설계하고 구현하는 개발자로 성장하고 있습니다. 

**핵심 역량 강화 방향**
- **클라우드 아키텍처 전문성**: AWS ECS, RDS, S3 등 고급 클라우드 서비스로 확장하여 대규모 시스템 운영 경험 축적
- **시스템 안정성 및 보안**: 모니터링, 백업, 보안 체계 구축을 통한 프로덕션 환경 운영 전문성 확보
- **팀 기술 리더십**: 협업 중심의 개발 문화 구축과 기술적 의사결정을 주도하는 개발자로 성장

**목표하는 개발자상**
안정적인 시스템을 책임지면서도, 비즈니스 요구사항을 기술적으로 해결하는 문제 해결 능력을 갖춘 개발자가 되는 것이 최종 목표입니다.
