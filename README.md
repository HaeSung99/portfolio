<details>
<summary><b>3. 가상의 스포츠 클래스 예약 플랫폼 (개인 프로젝트)</b> - 스포츠 클래스 신청 및 관리를 위한 풀스택 웹서비스</summary>

- **기간**: 2025.06.12 ~ 2025.07.08
- **팀 구성**: 1명 (풀스택)  
- **내 역할**: 풀스택 개발 전담

## 사용 기술
**Frontend**: `Next.js 15`, `TypeScript`, `Tailwind CSS`, `Axios`  
**Backend**: `NestJS`, `TypeScript`, `MySQL`, `TypeORM`, `Passport.js`, `JWT`, `bcrypt`, `Swagger`  
**DevOps**: `Docker`, `Docker Compose`

## 개요
구조화된 백엔드 프레임워크인 NestJS에 적응하고, 웹 서비스가 돌아가는 전체적인 흐름을 직접 구축하며 깊이 있게 이해하고자 기획한 프로젝트입니다.

단순히 이미 있는 사이트를 똑같이 따라 만드는 것에 그치지 않고, **'내가 실제 이 서비스를 운영하거나 이용하는 사람이라면 어떨까'**라는 고민을 하며 프로젝트 진행했습니다. 강사와 수강생의 입장에서 각각 어떤 기능이 필요할지를 중심에 두고 서비스를 설계했으며, 이 과정에서 발생하는 문제들을 해결하며 풀스택 개발자로서의 실무 역량을 길렀습니다.

## 주요 기능
1. 사용자 기능
   * JWT 기반의 보안 인증 시스템: JWT 기반 인증 시스템으로 안전한 회원가입 및 로그인 프로세스를 구현
     
   * 클래스 탐색 및 신청 시스템: 카테고리별 필터링을 통해 원하는 클래스를 빠르게 탐색할 수 있고, 클래스별 모집 상태(대기/모집/마감)를 실시간으로 반영하여 사용자에게 정확한 정보를 제공
     
2. 관리자 기능
   * 관리자 대시보드: 전체 클래스 신청자 통계 및 실시간 예약 현황을 한눈에 파악할 수 있는 관리자 대시보드를 구현
     
   * 데이터 관리 및 상태 동기화: 클래스 상태 변경 시 DB 내 상태값을 즉각 업데이트하여 데이터 정합성을 유지
     
3. 콘텐츠 운영 및 커뮤니케이션
   * 운영 알림 및 공지사항: 서비스 운영에 필요한 중요 공지사항을 작성하고 관리할 수 있는 게시판 기능을 구현
     
   * 활동 갤러리 관리: 클래스의 생생한 활동 사진을 업로드하고 관리할 수 있는 이미지 갤러리 기능을 제공

## 내 역할
* 프레임워크 전환 및 아키텍처 설계: 기존 익숙했던 Express 방식에서 벗어나, NestJS의 모듈형 구조를 도입하여 확장성 있는 백엔드 아키텍처 설계

* 데이터베이스 모델링 및 ORM 관리: TypeORM을 활용하여 사용자, 클래스, 신청 내역 간의 관계를 정의하고 데이터 정합성을 고려한 DB 스키마 설계

* 프론트엔드-백엔드 통합 개발: Next.js와 NestJS 간의 API 통신 규격을 정의하고 구현

* 도커 기반 개발 환경 표준화: Docker Compose를 사용하여 로컬 개발 환경과 DB 서버를 컨테이너화 함으로써 협업 상황 또는 실제 프로덕션 환경에서의 변수 대처

## 배운 점
* NestJS의 계층화된 아키텍처 이해: Module, Controller, Service로 분리된 구조를 직접 구현하며, 코드의 유지보수성과 가독성이 얼마나 향상되는지 체득

* 사용자 중심의 기능 우선순위 설정: 1인 개발로서 한정된 리소스 내에서 '신청 상태 관리'와 '관리자 대시보드' 등 서비스 운영에 필수적인 핵심 기능을 먼저 정의하고 구현하는 경험 체득

## 기억에 남는 이슈 해결 or 작업 경험
* 문제 상황: 여러 명의 관리자가 동시에 같은 클래스 신청 건을 승인/거절하거나 정보를 수정할 경우, 나중에 완료된 수정사항이 이전의 수정을 덮어쓰는 문제와 데이터 부정합이 발생할 가능성을 인지함

* 원인 분석: 단순한 Update 쿼리만으로는 데이터의 최종 상태를 보장할 수 없으며, 특히 신청 상태 변경과 같은 민감한 비즈니스 로직에서는 원자성뿐만 아니라 조회 시점과 수정 시점 사이의 데이터 변경 여부 확인이 필수적임을 파악함

* 해결 방법: 데이터의 원자성을 보장하기 위해 상태 변경과 로그 기록 로직을 하나의 트랜잭션으로 관리하였으며, 특히 낙관적 락을 도입하여 엔티티의 버전 번호를 통해 조회 시점 대비 데이터 변경 여부를 체크하도록 설계하여, 이를 통해 버전 충돌 시 예외를 발생시켜 부적절한 데이터 덮어쓰기를 원천 차단하고, 관리자에게 명확한 안내 메시지를 전달하여 동시 접속 환경에서도 완벽한 데이터 정합성을 유지

## 회고
기획부터 개발, 배포까지 서비스의 모두 담당하며 프론트엔드와 백엔드 전체를 관통하는 시스템의 흐름을 깊이 있게 이해할 수 있었습니다.

1인 프로젝트로서 프론트엔드와 백엔드를 동시에 진행하며 중심을 잡는 것이 쉽지 않았으나, 이를 통해 얻은 구조적인 사고방식은 단순히 기능을 구현하는 것을 넘어 서비스 전체의 유기적인 연결 구조를 파악하는 데 핵심적인 역량이 되었습니다. 이러한 통찰력은 앞으로 더 복잡하고 규모 있는 시스템을 마주했을 때도 전체적인 구조를 빠르게 파악하고, 문제를 정의하며 프로젝트를 안정적으로 완수할 수 있는 저만의 역량이 될 것이라 확신했습니다.

<summary><b>4. LostParty (팀 프로젝트 | 팀 리더)</b> - OCR 기반 실시간 게임 파티원 검증 및 스펙 조회 서비스</summary>

- **기간**: 2024.12 ~ 2025.04  
- **팀 구성**: 3명 (백엔드/프론트엔드/OCR)  
- **내 역할**: 백엔드 전담 + 팀 리더 

## 사용 기술
`Node.js`, `Express.js`, `MySQL`, `Docker`, `AWS EC2`, `Swagger`, `Git`, `Notion`  
프론트엔드 (팀원) : `Next.js` / OCR 모듈 (팀원) : `Python`

## 개요
OCR(광학 문자 인식) 기술과 게임 공식 API를 결합하여, 실시간 파티원의 정보를 재가공해 제공하고 커뮤니티형 블랙리스트를 공유할 수 있는 웹 서비스입니다.

일상적인 환경 속의 비효율을 기술로 해결하고자 기획했습니다. 수동 확인이 번거로웠던 파티원의 스펙 데이터를 사용자 중심의 UI로 재가공하여 가독성을 높였으며, 시스템상의 한계(차단 인원 제한, UI 알림 부재 등)로 인해 발생하는 게이머들의 실제 불편함을 커뮤니티 기반의 블랙리스트 시스템으로 보완하는 데 집중했습니다.

## 주요 기능
* 커뮤니티 기반 블랙리스트 서비스 구축: 유저 간 블랙리스트 공유 및 추천/비추천 시스템을 통한 데이터 신뢰도 확보
  
* 실시간 화면 분석 및 유저 식별: 브라우저 화면 공유 기능과 OCR을 활용하여 게임 화면 내 닉네임을 실시간으로 추출 및 검증
  
* 스펙 데이터 추출 및 재가공: 추출된 닉네임을 게임 공식 API와 연동하여, 파티 구성에 필요한 핵심 아이템 정보를 사용자 중심 UI로 재가공하여 제공
  
* 서비스 이용 통계 대시보드: 누적 가입자, 블랙리스트 등록 현황, 실시간 접속자 수 등 서비스 활성 지표 시각화

## 내 역할
* 서비스 아키텍처 및 데이터 플로우 설계: 아이디어 구체화부터 OCR 모듈, 백엔드, 게임 API 간의 데이터 파이프라인 전체 기획
  
* RESTful API 설계 및 백엔드 로직 구현: 블랙리스트 관리 로직, 추천 알고리즘 및 통계 수집 시스템 구축
  
* 데이터 전처리 및 큐레이션: API로부터 수신한 로우 데이터를 프론트엔드 요구사항에 최적화된 형태로 재가공하여 전달
  
* 관계형 데이터베이스(RDBMS) 설계: 효율적인 유저 검증과 데이터 정합성을 위한 MySQL 스키마 설계 및 인덱싱
  
* 클라우드 인프라 구축 및 배포: AWS EC2와 Docker를 활용하여 확장 가능한 배포 환경 구축

## 배운 점
* 협업에서의 소통의 중요성: 팀원 간 기술 스택 차이를 이해하고, 원활한 협업을 위한 API 명세서 및 데이터 인터페이스 커뮤니케이션의 중요성을 체득함
  
* 프로젝트 관리 역량: 팀 리더로서 팀원들의 작업 속도와 리소스를 고려하여 MVP 범위를 설정하고 일정을 관리하는 경험을 쌓음
  
* 실서비스 환경에 대한 변수 해결: 로컬 개발 환경과 실제 배포 환경의 차이점을 경험하고 해결하며, 환경 변수 및 보안 설정을 고려하는 역량을 쌓음

## 기억에 남는 이슈 해결
* 문제 상황: 로컬 개발 환경 에서는 정상 작동하던 화면 공유 기능이 AWS 배포 후 실제 배포 환경에서 작동하지 않음

* 원인 분석: 브라우저 화면 공유 기능은 HTTPS 환경에서만 작동하도록 제한되어 있었고, 로컬 환경은 예외적으로 허용되었으나, 실제 도메인 서비스에서는 SSL 인증서가 필수적이었음을 파악함

* 해결 방법: Nginx와 Certbot을 활용하여 SSL 인증서를 발급받고, 전체 서비스에 HTTPS 프로토콜을 적용하여 기능을 정상화함. 이를 통해 로컬 개발 환경과 다르게 웹 보안 정책이 실제 배포 환경에 미치는 영향력을 경험함

## 회고
당시 기획 단계에서 화면 공유 데이터를 실시간으로 백엔드에 전달하여 처리하는 방식을 고민했습니다. 하지만 그 당시 저의 웹소켓 기술 숙련도가 팀원과 함께 진행하는 프로젝트의 안정성을 담보하기에는 부족하다고 판단했고, 프로젝트의 성공적인 완수를 위해 보다 익숙하고 안정적인 구현 방식을 선택했습니다.

프로젝트를 마무리한 시점에서 돌아보니, 기술적 도전보다는 안전한 길을 선택했던 점이 UX 측면에서 아쉬움으로 남습니다. 만약 실시간 통신을 적용했다면 유저에게 훨씬 매끄러운 경험을 줄 수 있었을 것입니다. 이 아쉬움은 제가 이후 실시간 데이터 처리 기술을 더욱 깊게 공부하게 된 강력한 동기가 되었습니다.

# 기억에 남는 프로젝트
'로스트파티'는 게임 화면을 실시간으로 분석하여 파티원 정보를 자동 추출하는 서비스입니다. 초기 1인 개발로 시작했으나, 특수 폰트 인식 이라는 기술적 한계에 부딪혔습니다. 이를 해결하기 위해 OCR 기술 지식을 갖춘 팀원을 직접 모집하여 협업 체계로 전환했고, 팀을 통해 문제를 해결하는 과정을 경험했습니다.

평소 즐기던 게임의 불편함을 기술로 직접 해결하며 느낀 성취감은 제가 개발자로서 나아가야 할 방향성을 확고히 해주었습니다. 단순히 코드를 짜는 것에 그치지 않고, 사용자에게 실질적인 가치를 줄 수 있는 기능을 구현하기 위해 끊임없이 기술적 역량을 확장하고자 하는 동력을 얻은 프로젝트입니다.

# 다른 사람과 구별되는 나만의 강점
문제 발생 시 단편적인 해결책에 매몰되지 않고, 다양한 각도에서 분석하여 최선의 대안을 찾고 실행하는 능력이 저의 강점입니다.

특히 팀 프로젝트 진행 시, 모든 팀원의 의견을 공유하며, '학습적 가치'와 '개발 효율성'이라는 두 가지 관점에서 각 의견의 장단점을 객관적으로 비교 분석했습니다. 이러한 과정을 통해 팀 전체가 납득할 수 있는 합리적인 의사결정을 이끌어냈고, 이러한 부분에서 기술적으로 알지 못했던 사각지대 부분또한 팀원들로부터 많은 도움을 받아 학습했고, 원활한 협업 환경을 구축하는 커뮤니케이션 역량을 쌓아왔습니다.

# 성장 방향

**"기술과 비즈니스를 연결하는 개발자로 성장"**

풀스택 개발 역량과 클라우드 인프라 구축 경험을 고도화하여, 비즈니스 아이디어를 안정적인 서비스로 설계하고 구현해내는 개발자로 성장하고 싶습니다.

단순히 기술 스택을 쌓는 것에 만족하지 않고, 비즈니스 요구사항을 기술적 관점에서 현실적이고 효율적으로 풀어내는 '문제 해결사'가 되는 것이 저의 목표입니다. 기업의 목표를 이해하고, 이를 뒷받침할 수 있는 탄탄한 기술력을 제공하여 함께 성장하는 개발자가 되고 싶습니다.
